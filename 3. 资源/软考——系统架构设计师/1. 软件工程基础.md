# 2. 软件工程综述

## 2.1. 软件工程的定义


软件工程是指应用计算机科学、数学及管理科学等原理，以工程化的原则和方法来解决软件问题的工程，其目的是提高软件生产率、提高软件质量、降低软件成本。

## 2.2. 软件过程模型（软件生命周期模型）

软件要经历从需求分析、软件设计、软件开发、运行维护，直至被淘汰的全过程，这个过程称为软件生命周期。

### 2.2.1. 软件过程模型——定义

为了使软件生命周期中各项任务能够有序的按照规程进行，需要一定的工作模型对各项任务给予规程约束，这样的工作模型被称为软件过程模型，又称软件生命周期模型。    

### 2.2.2. 软件过程模型——瀑布模型

![[resource/Pasted image 20260528153832.png]]

瀑布模型包含的活动从一个阶段到另一个阶段逐次下降，工作流程在形式上很像瀑布。

- **特点**：因果关系紧密相连，前一个阶段的输出结果是后一个阶段输入。每个阶段完成后伴随一个里程碑，进行检查和确认。
- **适用**：需求很明确的项目。
- **优点**：有利于人员组织；便于管理。
- **缺点**：需求完整性、正确性很难确定；用户需要相当长的时间才能看到系统；纠错、变更代价巨大；很难在每个阶段一次性完全解决该阶段工作。

### 2.2.3. 软件过程模型——原型模型/快速原型模型

![[resource/Pasted image 20260528154056.png]]

- 包括**原型开发**、**目标软件开发**两个阶段。
- **途经**：模拟人机界面和交互方式；真正开发一个原型；找来类似软件进行比较。
- **类型**：
    - **抛弃型原型**：仅作为需求确认的手段，确认后即抛弃。
    - **演化型原型**：需求确认后不断补充完善，直到形成完整产品。
- **适用场合**：
	- 需求模糊不清；
	- 需要一定的开发环境和工具的支持；
	- 对原型的若干次修改后应收敛；
	- 大型软件难以设计原型。

### 2.2.4. 软件过程模型——螺旋模型

![[resource/Pasted image 20260528154314.png]]

- 原型模型与软件生命周期的结合。把软件开发流程分为多个阶段，每个阶段由4部分组成：：
    1. 目标设定
    2. 风险分析
    3. 开发及有效性验证
    4. 评审
- **特点**：**强调风险分析**，支持多种开发方法。
- **适用**：庞大、复杂且具有高风险的系统。
- **优点**：抵御风险能力强；支持需求变化。
- **缺点**：周期长；管理难度大；对风险分析能力要求高。

### 2.2.5. 软件过程模型——敏捷方法

- **总体目标**：通过尽早、持续的交付有价值的软件使客户满意。
- **适用**：团队和项目规模比较小。
- **特点**：
    - 适应性而非预设性
    - 面向人而非面向过程
    - 迭代增量式的开发

### 2.2.6. Scrum（敏捷开发框架）

核心：把大项目拆成短周期迭代，快速交付、持续调整。
1. 基本角色
- Product Owner（产品负责人）：定需求、排优先级
- Scrum Master（敏捷教练）：保障流程、扫清障碍、不做管理
- Development Team（开发团队）：跨职能、自组织，负责交付可用软件

2. 核心工件
- 产品待办列表（Product Backlog）：按商业价值排序的需求列表，包含所有想实现的功能、改进和修复
- Sprint 待办列表（Sprint Backlog）：本轮迭代要完成的任务
- 增量（Increment）：每个 Sprint 结束时可交付、可运行的产品版本

3. 核心事件
- Sprint（迭代）：固定周期，一般 2–4 周
- Sprint 计划会：确定本轮做什么、怎么做

每日站会：15 分钟内同步：昨天做了啥、今天做啥、有啥阻碍

- Sprint 评审会：向产品方演示可工作成果，收集反馈
- Sprint 回顾会：总结本轮问题，改进下一轮

### 2.2.7. 软件过程模型——统一过程（UP/RUP）

- **特点**：用例驱动，以体系结构为中心，迭代并增量。
- **四个阶段**：
    - 初始阶段：定义最终产品视图和业务模型，确定系统范围
    - 精化阶段：确定系统体系结构、制定工作计划及资源要求
    - 构建阶段：构造产品并继续演进
    - 移交阶段：产品发布
- **9个核心工作流**：业务建模、需求、分析与设计、实现、测试、部署、配置与变更管理、项目管理、环境。

## 2.3. 软件能力成熟度模型

- **初始级**：过程随意**混乱**，成功依赖个人能力，常超预算。
- **已管理级**：**项目级**过程有策划、文档化、执行、监督和控制，能实现成本、进度、质量目标。
- **已定义级**：**企业**定义适合自身和项目的标准流程，制度化，进行项目积累。
- **量化管理级**：建立产品质量、服务质量及过程性能的定量目标。
- **优化级**：通过增量式与创新式的过程与技术改进，不断改进过程性能。

## 2.4. 净室软件工程

- **理念**：通过严格的工程过程，在规约和设计阶段消除错误，避免依赖高成本的错误消除过程。
- **理论基础**：函数理论和抽样理论。
- **技术手段**：
    1. 统计过程控制下的增量式开发
    2. 基于函数的规范与设计
    3. 正确性验证
    4. 统计测试和软件认证
- **特点**：开发者不需要进行单元测试，而是进行正确性验证和统计质量控制。

## 2.5. 基于构件的软件工程

- **构件**：独立的软件单元，可与其它软件构成系统。
- **特征**：可组装性、可部署性、文档化、独立性、标准化。
- **组成**：接口、使用信息、部署。
- **CBSE主要过程**：系统需求概览 → 识别候选构件 → 根据发现的构件修改需求 → 体系结构设计 → 构件定制与适配 → 组装构件 → 创建系统。
- **构件组装方式**：顺序组装、层次组装、叠加组装。
- **构件不兼容**：参数不兼容、操作不兼容、操作不完备。

**Java EE 构件简介**

- **EJB（Enterprise JavaBeans）**：EJB是Java EE平台的一部分，用于开发企业级应用程序的服务器端构件。它们提供了事务管理、安全性、持久性、并发性和分布式对象访问等服务。
- **Servlets 和 JSP（JavaServer Pages）**：Servlets是Java类，用于扩展Web服务器的功能，而JSP是基于文本的表示层技术，用于动态生成HTML内容。它们经常一起使用来构建基于Java的Web应用程序。
- **JMS（Java Message Service）构件**：JMS是Java平台上消息传递的标准API，它允许应用程序创建、发送、接收和读取消息。JMS构件通常用于实现异步通信和分布式系统中的解耦。
- **JDBC（Java Database Connectivity）构件**：JDBC是Java应用程序与数据库之间进行通信的标准API。它允许Java程序执行SQL语句、检索结果和处理数据库中的错误。
  
**.NET 构件简介**

- **COM（Component Object Model）** 是微软提出的一个构件模型，它是一种以组件为发布单元的对象模型，这种模型使得各个软件组件可以用一种统一的方式进行交互。COM对象（CLSID）和COM接口（IID）由全局唯一标识符GUID所标识，是一个128位的整形值。用户在使用COM组件时，只能通过COM接口去操作，COM接口实际上是一个虚拟函数指针表，指向了拥有特定功能的实例的函数集。     


## 2.6. 软件开发环境与工具

### 2.6.1. 定义

软件开发环境是指支持软件的工程化开发和维护而使用的一组软件，由软件工具集和环境集成机制组成，前者用以支持软件开发的相关过程、活动和任务，后者为工具集成和软件的开发、维护及管理提供统一的支持。通过环境集成机制，各工具使用统一的接口或规范访问环境信息库，采用统一的界面形式。同时为各工具通信、切换、交流提供支持。

### 2.6.2. 集成机制

对工具的集成及用户软件的开发、维护及管理提供统一的支持，可划分为环境信息库、过程控制及消息服务器、环境用户界面三个部分。

- 环境信息库：是软件开发环境的核心，用以储存与系统开发有关的信息并支持信息的交流与共享。库中储存两类信息，一类是开发过程中产生的有关被开发系统的信息，如分析文档、设计文档、测试报告等；另一类是环境提供的支持信息，如文档模板、系统配置、过程模型、可复用构件等。

- 过程控制和消息服务器：是实现过程集成及控制集成的基础。过程集成是按照具体软件开发过程的要求进行工具的选择与组合，控制集成并行工具之间的通信和协同工作。

- 环境用户界面：包括环境总界面和由它实行统一控制的各环境部件及工具的界面。统一的、具有一致视感(Look & Feel)的用户界面是软件开发环境的重要特征，是充分发挥环境的优越性、高效地使用工具并减轻用户的学习负担的保证。   

### 2.6.3. 软件工具集

常分为软件开发工具、软件维护工具、软件管理和软件支持工具等。
 
软件开发工具：对应软件开发过程的各种活动，有需求分析工具、设计工具、编码与排错工具、测试工具。

需求分析工具：用以辅助需求分析活动，辅助系统分析人员从需求定义出发，生成完整的、清晰地、一致的功能规范。

分为基于自然语言或图形描述的工具和基于形式化需求定义语言的工具。    

编码工具：用以辅助程序员进行编码活动。
分为编辑程序、编译程序、汇编程序、生成程序。

排错工具：用以辅助程序员进行排错活动。
分为源代码排错程序、排错程序生成程序。  

软件维护工具：对应软件维护过程的各种活动，辅助维护人员对软件代码及文档进行各类维护活动。
分为版本控制工具、文档分析工具、开发信息库工具、逆向工程工具、再工程工具。

软件管理和软件支持工具：在软件全生命周期中，用以辅助软件管理和软件支持人员的管理活动和支持活动，保证软件高质量完成。
分为项目管理工具、配置管理工具、软件评价工具。

# 3. 需求工程

## 3.1. 需求与需求工程的定义

- **业务需求**：反映了组织机构或客户对系统、产品高层次的目标要求。
- **用户需求**：用户使用产品必须完成的任务。
- **功能/非功能需求**：定义了开发人员必须实现的软件功能，使用户能完成他们的任务，从而满足业务需求。还应包括非功能性需求，描述了系统展现给用户的行为和操作等，具体有产品必须遵从的标准、规范、合约；界面的细节；性能要求；设计或实现的约束条件、质量属性等。

需求工程是指应用已证实有效的原理、方法，通过合适的工具和记号，系统地描述待开发系统及其行为特征和相关约束。

通常可划分为如下阶段：
- **需求获取**：通过与用户的交流，对现有系统的观察及对任务进行分析，从而开发、捕获和修订用户的需求。
- **需求分析**：为系统建立概念模型，作为对需求的抽象描述，并尽可能多的捕获现实世界的语义。
- **形成需求规格**：又叫需求文档化，按照相关标准，生成需求模型的文档描述，作为后续开发的指南。
- **需求确认与验证**：以需求规格说明书为输入，通过用户确认、复审会议、符号执行、快速原型等方法，确认和验证需求规格的完整性、一致性、可测试性和可行性，包括有效性检查、一致性检查、可行性检查和确认可验证性。 
- **需求管理**：包括需求文档的追踪管理、变更控制、版本控制等活动。

需求最终文档经过评审批准后，则定义了开发工作的需求基线(baseline)。这个基线在客户和开发者之间构筑了计划产品功能需求和非功能需求的一个约定(agreement)。需求约定是需求开发和需求管理之间的桥梁。

## 3.2. 需求获取

需求获取是开发者、用户之间为了定义新系统而进行的交流，需求获取是获得系统必要的特征，或者是获得用户能接受的、系统必须满足的约束。

**需求获取的基本步骤**：开发高层的业务模型、定义项目范围和高层需求、识别用户角色和用户代表、获取具体的需求、确定目标系统的业务工作流、需求整理与总结。

**需求获取方法**：用户面谈、需求专题讨论会、问卷调查、现场观察、原型化方法、头脑风暴法。

## 3.3. 需求管理

需求管理是一个对系统需求变更、了解和控制的过程。一旦形成了需求文档初稿就开始了需求管理活动。

## 3.4. 需求变更

需求变更是不可避免的，但必须是受控的。

变更管理的主要内容：仔细评估已建议的变更；选择合适人员评估变更；变更应及时通知相关人员；按一定流程处理变更，对变更的过程和状态进行控制。

变更控制过程：

![[resource/Pasted image 20260528171120.png]]

常见变更策略：
- 所有变更必须遵循变更控制流程；
- 未获批准的变更，不应做设计和实现工作；
- 变更应该由变更控制委员会决定实现；

- 决不能从配置库删除或修改变更请求的原始文档；
- 每一个集成的需求变更必须能跟踪到一个经核准的变更请求；

变更控制的执行机构——变更控制委员会(CCB)

## 3.5. 需求追踪

需求跟踪包括编制每个需求同系统元素之间的联系文档，这些元素包括其它需求、体系结构、其他设计部件、源代码块、测试、帮助文件和文档等，是要在整个项目的工件之间形成水平可追踪性。

需求跟踪有两种形式：正向跟踪、逆向跟踪；

二者合称双向跟踪，通过建立需求跟踪矩阵来实现。

# 4. 软件测量与度量

## 4.1. 软件测试概述

软件测试是使用人工或自动的手段来运行或测定某个软件系统的过程，其目的在于检验它是否满足规定的需求或弄清预期结果与实际结果间的差别。

**按测试中程序执行的状态分为**：静态测试、动态测试

**按是否测试系统内部分为**：黑盒测试、白盒测试、灰盒测试

**按程序的执行方式分为**：人工测试、自动化测试

软件调试：定位并修复已经发现的缺陷。

基本原则：先测试，后调试

![[resource/Pasted image 20260528171626.png]]

## 4.2. 软件测试的方法

### 4.2.1. 黑盒测试

黑盒测试也成为功能测试，在完全不考虑软件内部结构和特性的情况下，测试软件的外部特征。

**测试方法**：等价类划分、边界值分析、错误推测、因果图

**等价类划分**：将程序的输入域划分为若干等价的类，然后从每种等价的类中选取一个代表性数据作为测试用例。

**边界值分析**：输入的边界比中间更易发生错误，用边界值分析补充等价类划分。

### 4.2.2. 白盒测试

白盒测试也称为结构测试，根据程序的内部结构和逻辑来设计测试用例，对程序的路径和过程进行测试，检查是否满足设计的需要。

**测试方法**：逻辑覆盖、循环覆盖、基本路径测试

**语句覆盖**：选择足够的测试数据，使被测程序中每条语句至少执行一次。很弱的逻辑覆盖。

**判定覆盖**：选择足够的测试用例，使得被测程序中的每个判断表达式至少获得一次真值、一次假值。

**条件覆盖**：选择足够的测试用例，使得被测程序中的每个逻辑条件的各种可能取值至少执行一次。

**路径覆盖**：覆盖被测试程序中所有可能路径。

### 4.2.3. 测试自动化

测试自动化主要以脚本技术来生成测试用例。脚本是一组可供测试工具执行的指令的集合。

脚本可分为：线性脚本、结构化脚本、共享脚本、数据驱动脚本、关键字驱动脚本。

# 5. 软件维护 

## 5.1. 软件维护的内容

- **正确性维护**是指改正在系统开发阶段已发生而系统测试阶段尚未发现的错误。
 
- **适应性维护**是指使应用软件适应信息技术变化和管理需求变化而进行的修改。

- **完善性维护**是指为扩充功能和改善性能而进行的修改，主要是指对已有软件系统增加一些在系统分析设计阶段没有的功能与性能特征。

- **预防性维护**是指为了改进应用软件的可靠性和可维护性，为了适应未来的软/硬件环境的变化，主动增加预防性的新功能。

## 系统转换与交接

![[resource/Pasted image 20260602095937.png]]

**直接转换**：旧系统停止，新系统立即运行，无过渡阶段。人力和费用最省，风险大，适用于新系统不太复杂或现有系统完全不能用的场景。  

![[resource/Pasted image 20260602100001.png]]

**并行转换**：新系统和现有系统并行工作一段时间，试运行结束后再切换。首先以旧系统作业为正式作业，新系统校核用，经过一段时间，待新系统测试稳定后再切换。风险小，人力费用消耗大。

![[resource/Pasted image 20260602100141.png]]

**分段转换**：直接转换和并行转换的结合，采用分期分批逐步转换，可按功能、按部门、按机器设备分功能转换。适用于比较大的系统；或旧系统比较稳定，能适应业务需要；或转换风险大。震动小，用户容易接受，但周期更长，人力费用消耗更大。

## 5.3. 文档管理

**用户文档**描述系统功能和使用方法，并不关心技术实现。 

**系统文档**描述系统设计、实现和测试等各方面的内容。

- 文档是软件可维护性的决定因素，必须完整。
- 维护阶段对软件的任何修改都必须反映在用户文档和系统文档中
